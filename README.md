# Lily Labs — Compiler Tests in the Open

> Every test we run against our compiler. In public. Every day.

This repository runs the **full test suite** against the
currently-published version of the Lily Labs compiler (`lly`)
from our Release Channel (`rc.lilylabs.io`). Results are
committed daily, the workflow is open, the corpora are open,
the numbers are honest.

If we regress, you will see it here before we do.

## Why this exists

Building a polyglot compiler is hard. Building one that the
world can trust requires showing your work. We benchmark
against the industry-standard correctness suites — **test262,
c-testsuite, GCC torture, GCC dg, LLVM single-source** — and
publish the pass rates without filtering.

This repository is the receipt.

## What gets tested

We compile the official test corpora of the languages we
support, run the binaries against expected outputs, and
report the pass rate. Everything reproducible, everything in
the open.

| Suite | Compiler | Source corpus | Baseline | What it proves |
|---|---|---|---|---|
| **c-brick** | machineroom (C) | self-written | 100% | C → WASM-brick roundtrip works |
| **cpp-brick** | machineroom (C++) | self-written | 100% | C++ → WASM-brick roundtrip works |
| **brick-format** | machineroom | self-written | 100% | Brick manifest format is stable |
| **realworld-build** | machineroom | curated real-world | 100% | Real C/C++ programs (jq, hello-world tooling, etc.) build & smoke-pass |
| **c-testsuite** | machineroom | [c-testsuite/c-testsuite](https://github.com/c-testsuite/c-testsuite) | 99.5% | Established C-correctness suite |
| **gcc-torture** | machineroom | [gcc-mirror/gcc](https://github.com/gcc-mirror/gcc) `gcc.c-torture` | 85.6% | GCC's own correctness torture-tests |
| **gcc-dg-cpp** | machineroom | [gcc-mirror/gcc](https://github.com/gcc-mirror/gcc) `g++.dg` | 83.9% | GCC's C++ diagnostics suite |
| **llvm-single** | machineroom | [llvm/llvm-test-suite](https://github.com/llvm/llvm-test-suite) | 64.0% | LLVM single-source correctness |

Each suite has its own [docs/suites/<name>.md](docs/suites/)
explaining what it tests, why it matters, and the
provenance of the test inputs.

## Methodology — what we measure and what we don't

We test the **published** compiler — the one anyone can
install via `curl rc.lilylabs.io/install.sh | bash`.

- **No mocks, no skips, no "best of 3" runs.** Every
  test is executed against the latest binary, results are
  reported as-is.
- **External corpora pinned by commit-SHA.** When upstream
  test262 changes, we manually re-pin, document the change,
  and explain any pass-rate movement.
- **Failures get committed too.** Reports include the
  list of failing tests — not just an aggregate number.
- **Trend over time matters more than any single number.**
  See the [trend page](https://lilylabs.github.io/compiler-tests/)
  for our history.

See [docs/methodology.md](docs/methodology.md) for the full
methodology — including what we deliberately don't measure
and why.

## What the workflow does

```
┌──────────────────────────────────┐
│ 1. install lly from RC (≈15s)    │
│    curl install.sh | bash         │
├──────────────────────────────────┤
│ 2. download plugins from RC      │
│    (c, cpp, brick-format, …)      │
├──────────────────────────────────┤
│ 3. restore external corpora       │
│    (test262, gcc, llvm — cached)  │
├──────────────────────────────────┤
│ 4. run each suite                 │
│    npm start -- --suite <id>      │
├──────────────────────────────────┤
│ 5. emit JSON report               │
│    commit to results/YYYY-MM-DD   │
├──────────────────────────────────┤
│ 6. update badges + trend page     │
└──────────────────────────────────┘
```

No source-build, no SSH-keys, no internal access. The
workflow runs against the same binaries the public installs.

## Trigger the workflow yourself

If you've forked this repo (or if you have access),
**Actions → Test Compiler → Run workflow**, or via CLI:

```bash
gh workflow run test-compiler.yml \
  --field suites="c-brick,cpp-brick,brick-format,realworld-build" \
  --field plugins="c cpp brick-format"
```

Inputs:

| Input | Default | What |
|---|---|---|
| `tier` | `smoke` | `smoke` / `nightly` / `weekly` — preset suite groups |
| `suites` | (from tier) | override: comma-separated suite IDs or `all` |
| `plugins` | `c cpp nextjs deploy brick-format` | which `lly plugin`s to install |
| `rc_url` | `https://rc.lilylabs.io` | override to test pre-release RC |

This is **not** the compiler itself. The Lily Labs
compiler (`machineroom`, `oldies/nex`) lives in a
separate repository.

This is what runs every night against the published
binary. Think of it as the public Quality-Assurance log.
