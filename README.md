# Lily Labs — C/C++ Compiler Tests in the Open

> Every C and C++ test we run against our compiler. In public.
> Triggered → runs the full suite. No tiers, no shortcuts.

This repository runs the **complete C and C++ test suite**
against the currently-published version of the Lily Labs
compiler (`lly`) from our Release Channel (`rc.lilylabs.io`).

Trigger the workflow, the whole thing runs end-to-end:
self-written tests, the official `c-testsuite`, GCC's own
torture tests, GCC's C++ diagnostics, and the LLVM
single-source corpus. Pass-rate gets committed, badges
update, trend page redraws.

If we regress, you'll see it here before we do.

## Why this exists

Building a polyglot compiler is hard. Building one that the
world can trust requires showing your work. We benchmark
against the industry-standard correctness corpora —
**c-testsuite, GCC torture, GCC dg, LLVM single-source** —
and publish the pass rates without filtering.

This repository is the receipt.

## Why C and C++ only

The compiler supports more languages (JavaScript, Next.js
build pipeline, others on the way), but their test
infrastructure has different needs that don't belong in
this public correctness CI:

- **JavaScript** — testbench for test262/wpt needs the
  legacy `nex` binary which isn't on the public RC yet
- **Next.js E2E, React-SSR, cap-http** — need full
  cloud-stack access (CAPs, control-plane), which a
  public repo can't have
- **Rust, Go frontends** — not yet stable enough to
  benchmark publicly

C and C++ are the most mature frontends and have the most
established external test corpora. They're where the
trust-signal is highest and the measurement is cleanest.

## Methodology — what we measure and what we don't

We test the **published** compiler — the one anyone can
install via `curl rc.lilylabs.io/install.sh | bash`.

- **No mocks, no skips, no "best of 3" runs.** Every
  test is executed against the latest binary, results are
  reported as-is.
- **External corpora pinned by commit-SHA.** When upstream
  GCC/LLVM/c-testsuite change, we manually re-pin and
  document any pass-rate movement.
- **Failures get committed too.** Reports include the
  list of failing tests — not just an aggregate number.
- **Trend over time matters more than any single number.**
  See the [trend page](https://llylabs.github.io/compiler-tests/)
  for our history.

See [docs/methodology.md](docs/methodology.md) for the
full methodology — including what we deliberately don't
measure and why.

## What the workflow does

```
┌──────────────────────────────────┐
│ 1. install lly from RC           │
│    (anonymous via install.sh)    │
├──────────────────────────────────┤
│ 2. download c + cpp + brick      │
│    plugins from RC (via OIDC,    │
│    no secrets in the repo)       │
├──────────────────────────────────┤
│ 3. restore external corpora      │
│    (c-testsuite, gcc, llvm —     │
│    cached across runs)            │
├──────────────────────────────────┤
│ 4. run ALL 8 C/C++ suites        │
│    sequentially                   │
├──────────────────────────────────┤
│ 5. emit JSON report               │
│    commit to results/YYYY-MM-DD  │
├──────────────────────────────────┤
│ 6. update README badges           │
│    redeploy trend page (Pages)   │
└──────────────────────────────────┘
```

No source-build, no SSH keys, no internal access. The
workflow runs against the same binaries the public installs.

## Triggers

The workflow runs the **same full C/C++ suite** every time
— no tier presets, no smoke runs, no shortcuts:

| Trigger | When |
|---|---|
| `workflow_dispatch` | manually, via Actions UI or `gh workflow run` |
| `schedule` cron `0 2 * * *` | nightly at 02:00 UTC |
| `push` to `main` | when `testbench/`, `scripts/`, or workflow changes |

Manual trigger:

```bash
gh workflow run test-compiler.yml --repo llylabs/compiler-tests
```

Optional input `rc_url` lets you test against a staging RC
instead of production.

## What this repo is *not*

This is **not** the compiler itself. The Lily Labs
compiler (`machineroom`) lives in a separate repository.

trigger, every night. Think of it as the public
Quality-Assurance log for C and C++ correctness.
